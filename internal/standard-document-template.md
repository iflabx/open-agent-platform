# 标准页统一模板

> 用途：标准说明页编写模板
>
> 使用范围：`docs/standards/*.md`
>
> 发布规则：本文件仅供内部编写使用，不作为公开页面发布

## 一、编写目标

标准页要回答的不是“这个标准的所有历史细节”，而是：

`这个标准约束什么，它在当前方案里落在哪些链路上，它不能替代什么。`

因此，标准页优先写：

- 当前公开口径
- 标准约束对象
- 在当前方案中的落点
- 与组件和协议的关系
- 采用规则和治理要求

不优先写：

- 冗长的历史沿革
- 与当前方案无关的扩展讨论
- 大量实现细节或安装步骤

## 二、统一结构

每个标准页建议按以下结构编写：

- `类型`
- `当前口径`
- `当前定位`
- `标准约束什么`
- `核心内容`
- `在当前方案中的落点`
- `与组件 / 协议关系`
- `适合场景`
- `边界`
- `采用规则`
- `治理注意点`
- `关联文档`
- `参考资料`

## 三、模板

```md
# 标准名称

> 类型：接口兼容 / 数据结构 / 运行语义 / 其他标准类型
>
> 当前口径：截至 YYYY-MM-DD，当前采用的版本、草案或事实标准口径

## 当前定位

用一段话说明这个标准在当前方案中的角色。

## 标准约束什么

- 约束对象 1
- 约束对象 2
- 约束对象 3

## 核心内容

- 核心概念 1
- 核心结构 2
- 核心规则 3

## 在当前方案中的落点

- 落点 1
- 落点 2
- 落点 3

## 与组件 / 协议关系

- 与组件的关系
- 与协议的关系
- 与其他标准的关系

## 适合场景

- 场景 1
- 场景 2

## 边界

- 不能替代什么
- 不负责什么
- 不建议如何滥用

## 采用规则

- 当前方案采用的范围
- 版本、草案或兼容口径
- 哪些能力必须统一，哪些能力可以条件扩展

## 治理注意点

- 版本管理
- 兼容性管理
- 安全、审计或观测要求

## 关联文档

- [标准页总览](/standards/)
- [相关主题页](/protocol-standards)
- [相关组件页](/components/example)

## 参考资料

- [官方规范或标准页](https://example.com)
- [官方文档](https://example.com/docs)
- [官方仓库或项目页](https://github.com/example/example)
```

## 四、写作要求

- 对正式标准、草案和事实标准要明确区分，不混写。
- `当前口径` 必须给出当前可依赖的版本、草案或兼容基线。
- 标准页要写“约束和边界”，不退化成“某个组件支持这个标准”的说明页。
- 如果某项标准在当前方案中只是跨层配套，不要把它写成单层专属能力。
- 参考资料统一放在页面底部，并优先使用官方一手来源。
