## 一、这页要解决什么问题

企业智能体方案里，常见问题不是“有没有协议”，而是容易把几类东西混在一起：

- 真正的运行时互操作协议
- 身份与接入标准
- API 与工具描述规范
- 模型调用接口规范
- 观测与链路语义标准
- 授权与治理方法

如果这些不分层，后续在写方案、做平台、接系统时就会混乱。

所以当前方案把“协议与标准体系”分成六层：

1. 智能体互操作协议
2. 身份与组织接入标准
3. API 与工具描述标准
4. 模型服务接口标准
5. 观测与运行语义标准
6. 授权与治理规范

## 三、智能体互操作协议

这是最接近“智能体协议层”的一组标准。

### 1. MCP

- 定位：agent-to-tool / context
- 作用：统一工具、资源、提示词、上下文能力接入
- 当前方案角色：主平台、自研 Agent、知识服务、外部系统工具之间的标准连接层
- 边界：不负责 agent-to-agent 协作

详细见 [MCP 协议](/protocols/mcp)。

### 2. A2A

- 定位：agent-to-agent
- 作用：远程 Agent 发现、调用、任务传递与协作
- 当前方案角色：跨系统、跨团队、多 Agent 分工场景的标准协作协议
- 边界：不替代工具接入协议

详细见 [A2A 协议](/protocols/a2a)。

### 3. 当前方案中的边界划分

- `MCP` 负责接工具、资源和上下文
- `A2A` 负责接远程 Agent
- 不用 `MCP` 替代远程 Agent 协作
- 不用 `A2A` 替代工具接入

## 四、身份与组织接入标准

这一层不是“智能体协议”，但它决定企业平台能不能真正落地。

### 1. OIDC / OAuth2 / SAML / SCIM

- `OIDC / OAuth2`：适合统一登录、访问令牌、应用接入
- `SAML`：很多传统企业系统仍在用，尤其在老 IAM 体系里常见
- `SCIM`：适合用户、组织、组、角色等身份对象同步

当前方案里的定位：

- 不单独自建身份产品
- 默认对接企业现有 `SSO / IAM`
- 门户、BFF、知识服务、工具服务都继承统一身份上下文

这层回答的是：用户是谁、属于哪个部门、带什么组织属性进入平台。

详情页入口见 [身份接入协议（OIDC / OAuth2 / SAML / SCIM）](/protocols/identity-access)。

## 五、API 与工具描述标准

这一层解决的是“接口怎么描述”“参数怎么约束”“系统怎么对接”。

### 1. OpenAPI

- 用于描述 HTTP / REST 接口契约
- 适合门户 BFF、平台服务、工具服务和外部业务系统集成
- 有助于统一 API 文档、SDK 生成和接口治理

### 2. JSON Schema

- 用于定义输入输出结构、字段类型、校验规则和参数约束
- 适合工具调用参数描述、结构化结果定义、表单类任务校验
- 在 `MCP`、工具层和服务契约层都很重要

这一层回答的是：接口长什么样、参数是否合法、系统如何稳定对接。

详情页入口见 [OpenAPI](/protocols/openapi) 和 [JSON Schema](/standards/json-schema)。

## 六、模型服务接口标准

### 1. OpenAI 兼容 API

当前方案没有把模型接口做成私有格式，而是统一收敛到 `OpenAI 兼容 API`。

当前方案里的定位：

- `LiteLLM` 对上提供统一兼容接口
- 上层平台与自研服务优先按兼容接口接入
- 底层模型和推理后端变化不直接传导到上层业务

这一层回答的是：上层应用如何稳定调用模型，而不绑定某一家底层实现。

详情页入口见 [OpenAI 兼容 API](/standards/openai-compatible-api)。

## 七、观测与运行语义标准

### 1. OpenTelemetry

这不是“监控工具”，而是统一埋点、上下文传播和 Trace 语义标准。

当前方案里的定位：

- 统一 `trace_id / session_id / app_id / user_id / tenant_id`
- 把入口网关、门户、编排、检索、模型、工具调用串成一条链
- 与 `Prometheus + Grafana + Loki + LangFuse` 一起形成统一观测体系

这一层回答的是：一次请求从入口到模型到工具到返回，是否能被完整追踪。

详情页入口见 [OpenTelemetry](/standards/opentelemetry)。

## 八、授权与治理规范

这部分严格说不全是“协议”，但在企业架构里必须和协议层并列看待。

### 1. RBAC + ABAC

- `RBAC`：基于角色的授权
- `ABAC`：基于部门、项目、数据域、标签、租户等属性授权

当前方案里的定位：

- 由 `Casbin` 承担统一裁决能力
- 应用访问、知识检索、工具调用、数据标签都纳入完整授权链路

### 2. Policy as Code

- 把策略写成受控配置，而不是散落在代码里的 if/else
- 支持版本化、审批、审计、发布和回滚
- 适合权限策略、内容审查策略、风险控制策略

### 3. Maker-Checker

- 即人工审批 / 双人复核机制
- 适用于写数据、发消息、改权限、外部生产系统操作等高风险动作
- 在当前方案中已经上升为架构强约束，不是“可选能力”

这一层回答的是：谁能做什么、策略怎么变更、高风险动作谁来兜底。

详情页入口见 [RBAC + ABAC](/norms/rbac-abac)、[Policy as Code](/norms/policy-as-code)、[Maker-Checker](/norms/maker-checker)。

## 九、当前方案的协议与标准落点

| 层 | 主要标准 / 协议 | 当前承载方式 |
| --- | --- | --- |
| 入口接入层 | `OIDC / OAuth2 / SAML / SCIM` | 企业 `SSO / IAM` + `APISIX` + 门户/BFF |
| 门户与服务接口层 | `OpenAPI`、`JSON Schema` | 门户、BFF、工具服务、自研服务契约 |
| 智能体互操作层 | `MCP`、`A2A` | 工具接入、上下文接入、远程 Agent 协作 |
| 模型服务层 | `OpenAI 兼容 API` | `LiteLLM` 对上统一接口 |
| 观测层 | `OpenTelemetry` | 全链路埋点、Trace 语义与上下文传播 |
| 授权治理层 | `RBAC + ABAC`、`Policy as Code`、`Maker-Checker` | `Casbin` + 门户/BFF/知识层/工具层执行点 |

## 十、当前未正式纳入、但后续可关注的标准

这部分当前方案还没有正式列为主清单协议，但后续如果异步作业和事件化能力增强，可以再评估：

- `AsyncAPI`
- `CloudEvents`

它们更偏向异步接口描述和事件标准，不属于当前方案的核心必选项。

## 十一、一句话收敛

当前方案里，真正的智能体协议层是 `MCP + A2A`；支撑企业落地的外围标准则包括：

- 身份接入：`OIDC / OAuth2 / SAML / SCIM`
- 接口描述：`OpenAPI`、`JSON Schema`
- 模型接口：`OpenAI 兼容 API`
- 观测语义：`OpenTelemetry`
- 授权治理：`RBAC + ABAC`、`Policy as Code`、`Maker-Checker`

只有把这些一起看，方案才算完整。
