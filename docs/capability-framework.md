## 一、这个文档和“组件 / 协议页面”的区别

两者不是一回事。

- 组件 / 协议页面回答的是：当前方案某一层可以采用哪些具体实现项。
- 能力与规范体系文档回答的是：一家企业级智能体平台，无论采用什么产品，都必须具备哪些能力、遵循哪些约束、达到什么完整性。

前者偏“实现项与落地路径”，后者偏“能力与规范基线”。

因此，这里会出现三类内容：

- 明确的能力项
- 应满足的规范和约束
- 当前方案中可对应承载这些能力的组件

需要特别明确四点：

1. 能力规范不要求一一对应某个开源软件或协议。
2. 有些能力本身就是理念、流程、治理要求或架构约束，没有单独组件。
3. 一个能力可能由多个组件共同承载，一个组件也可能支撑多个能力域。
4. 组件 / 协议页面属于“可实施项”，只有能够落到方案某一层的具体实现，才应该进入组件页体系。

组件 / 协议进入正式方案清单前，应满足以下准入原则：

- 可以免费使用
- 许可证清晰
- 法律风险可控
- 适合私有化企业场景

如果某个项目存在许可证边界、商用限制或法务不确定性，它可以出现在研究结论或开源状态说明里，但不应被轻易视为正式稳定推荐项。

## 二、总体原则

企业智能体平台应遵循以下原则：

1. 平台化优先于项目化，不为单个场景重复建设底座。
2. 控制面、运行面、数据面、治理面必须分清，不把所有职责堆在一个应用里。
3. 所有模型调用、知识检索、工具执行、跨 Agent 协作都必须可追踪、可审计、可回放。
4. 高风险动作必须引入人工审批、人工接管或双人复核，不允许默认全自动。
5. 权限控制必须是完整授权链路，而不是某个系统里“有一份角色表”。
6. 观测体系必须采用统一语义，而不是日志、指标、Trace 各自割裂。
7. Prompt、工具、知识、模型路由、策略规则都属于受控配置，必须版本化、可回滚、可灰度。
8. 协议优先于私有耦合，优先采用开放协议和标准接口。

## 四、详细能力与规范要求

### 1. 身份、组织与访问上下文

企业智能体平台不是匿名聊天系统，必须带组织上下文运行。

最低要求：

- 所有用户都通过企业统一身份体系进入平台
- 平台能识别用户、部门、岗位、租户、项目或数据域
- 用户上下文能传递到检索层、工具层、Agent 层和审计层
- 服务账号、机器人账号、系统间调用身份要单独管理

规范要求：

- 优先对接企业现有 `OIDC / OAuth2 / SAML / SCIM`
- 用户身份和组织属性不能只存在前端 Session，必须传递到后端运行时
- 身份与授权分离，认证不等于授权

当前方案对应组件：

- 企业现有 `SSO / IAM`
- 门户层 `AgentifUI`
- 授权链路 `Casbin`

### 2. 门户、应用目录与应用分发

企业智能体平台需要“统一入口”，而不是多个零散聊天页面。

最低要求：

- 提供统一门户
- 提供应用目录、应用搜索、应用分组和应用 owner 信息
- 用户可以区分公共应用、部门应用、个人应用
- 反馈、工单、问题回报、满意度采样要进入运营闭环

规范要求：

- 每个应用都应有唯一标识、归属团队、版本、风险等级、发布状态
- 门户只负责入口和交互，不持有核心编排逻辑

当前方案对应组件：

- `APISIX`
- `AgentifUI`
- 自研 `BFF / API` 接入层

### 3. 模型服务与模型治理

模型必须被当成受控基础设施，而不是任何业务都能直接连。

最低要求：

- 统一模型入口
- 模型路由、配额、限流、故障切换
- 模型版本记录、模型上线审批和回滚能力
- 输入输出留痕，至少保留摘要级审计

规范要求：

- 统一兼容接口，避免上层直接耦合某个模型厂商
- 模型卡、能力边界、适用场景、风险等级要有登记
- 代码模型、安全审查模型、通用模型应拆池部署

当前方案对应组件：

- `LiteLLM`
- `vLLM`
- `Qwen3.5 27B FP8`
- `Qwen3-Coder-Next`
- `Qwen3Guard-Stream-8B`

### 4. Agent 运行时与编排模式

企业级 Agent 不只是“Prompt + LLM”，而是有状态、有边界、有恢复能力的运行时。

最低要求：

- 能表达多步任务、工具调用、分支判断和状态持久化
- 能处理失败恢复、超时、重试、补偿和人工接管
- 编排模式要被正式建模，而不是靠临时代码拼接

正式编排模式建议纳入方案标准：

1. 顺序编排：适合标准问答、表单补全、单链路流程
2. 并发编排：适合多知识源召回、多专家并行分析、多工具并行执行
3. 交接编排：适合服务台转派、领域 Agent 分工、远程 Agent 协作
4. 监督者-执行者编排：适合 Planner-Worker、任务拆解与汇总
5. 审批/复核编排：适合高风险写操作、流程审批、外部通知、制度性动作
6. 异步作业编排：适合长时任务、批量索引、批处理报告和回调任务

规范要求：

- 高风险动作必须进入审批/复核模式，不能走全自动直写
- 长时任务必须支持持久化和恢复
- 编排逻辑、工具 Schema、上下文模板要版本化

当前方案对应组件：

- 标准场景：`Dify`、`RAGFlow`、`Coze Studio`
- 自研复杂编排：`LangChain`
- 生产级状态运行时：`LangGraph`
- 长期记忆场景：`Letta`

### 5. 工具接入、协议互操作与系统集成

企业智能体平台必须解决“怎么接工具”和“怎么接其他 Agent”这两件事。

最低要求：

- 工具接入有统一接口，不让每个应用自定义私有调用方式
- 工具描述、参数、返回结构有 Schema
- 支持跨系统、跨团队的 Agent 协作

规范要求：

- `MCP` 负责 agent-to-tool / context
- `A2A` 负责 agent-to-agent
- 工具风险等级、参数校验、幂等键、超时与重试必须标准化
- 工具调用日志必须纳入统一 Trace

当前方案对应组件：

- `MCP`
- `A2A`

### 6. 数据治理层与企业 AI 数据底座

企业 AI 平台不应让每个应用直接连接原始系统，而应先经过统一数据治理层。

最低要求：

- 统一登记数据源、责任人、更新频率和使用边界
- 同时覆盖结构化与非结构化数据
- 数据先进入治理流程，再供给给 `RAG / 搜索 / 查询 / Agent`
- 元数据、权限标签、生命周期状态能够被统一管理

规范要求：

- 数据应至少区分 `Raw / Staged / Curated` 三层
- 所有 AI 系统优先消费治理后的数据产品，而不是直接连原始源系统
- 权限标签、分类分级、时效与失效状态必须进入 AI 检索和查询链路
- 统一元数据和资产目录应覆盖结构化与非结构化数据

当前方案对应组件：

- 元数据治理中枢：`OpenMetadata`
- 数据采集与同步：`SeaTunnel`
- 结构化数据治理：`dbt Core`
- 非结构化解析：`Apache Tika`
- 原始与对象数据承载：`MinIO`
- 元数据与配置承载：`PostgreSQL`
- 知识工程与索引编排：`LlamaIndex`
- 平台内知识接入：`Dify`、`RAGFlow`、`Coze Studio`
- 详细设计见 [数据治理层（企业 AI 数据底座）](/data-governance)

### 7. 知识与 RAG 标准链路

企业 RAG 不是“把文件丢进向量库”这么简单，必须是完整知识链路。

标准链路建议至少包含：

1. 数据源登记：明确来源系统、责任人、同步方式、更新频率
2. 文档接入：文件、Wiki、数据库、工单、FAQ、API 等多源接入
3. 解析与清洗：OCR、结构抽取、格式清洗、去重、去噪
4. 切片与元数据：章节、标题、标签、来源、时效、权限标签
5. 权限注入：部门、角色、项目、数据域标签写入索引元数据
6. Embedding：统一向量化模型
7. 索引构建：向量索引、全文索引或混合索引
8. 查询改写：意图规范化、query expansion、上下文补充
9. 召回：向量召回、关键词召回、混合召回
10. 权限过滤：先过滤可访问范围，再交给重排
11. 重排：按语义相关性、业务规则、文档时效做重排
12. 回答生成：带引用、带来源、带置信提示
13. 输出审查：安全、合规、敏感信息与越权检查
14. 反馈闭环：用户反馈、低质量样本回收、重建索引

规范要求：

- 检索必须是权限感知的
- 回答必须尽可能带来源引用
- 知识库必须支持增量同步和失效处理
- 不允许把“全库原文”默认塞给模型

当前方案对应组件：

- 知识中枢：`Dify`、`RAGFlow`、`Coze Studio`
- 知识工程：`LlamaIndex`
- Embedding / Rerank：`Qwen/Qwen3-Embedding-8B`、`Qwen/Qwen3-Reranker-8B`
- 存储底座：`MinIO`
- 检索底座：`Weaviate`、`Milvus`、`Elasticsearch`

### 8. 控制面与发布治理

这是企业方案里最容易缺失、但最关键的一层。

控制面不是某个单独产品，而是一组必须存在的平台能力。

最低要求：

- 应用注册与应用目录
- Prompt 版本管理
- 工具 Schema 版本管理
- 知识库版本与索引版本管理
- 模型路由与策略配置管理
- 评测准入、灰度发布、回滚与变更审计

规范要求：

- 所有影响行为的配置都应被视为“受控配置”
- 配置变更必须有版本、审批、灰度和回滚
- 应用上线前至少经过离线评测和小流量验证
- 不允许在生产直接手工改 Prompt、Tool、知识索引关键配置

当前方案对应组件：

- 平台自研控制面
- `LangFuse` 承担评测与回放能力
- Git 仓库与发布流程承担版本化能力

### 9. 完整授权链路与安全治理

企业权限体系不能只写“我们用 Casbin”就结束，必须是完整链路。

建议最少包含四类角色：

- `PAP`：策略管理点，负责策略定义、发布、版本和审批
- `PDP`：策略决策点，负责实时授权判断
- `PEP`：策略执行点，负责在请求入口、检索层、工具层执行拦截
- `PIP`：策略信息点，负责提供用户属性、数据标签、组织信息、上下文事实

最低要求：

- 应用访问授权
- 知识检索授权
- 工具调用授权
- 数据标签与租户隔离
- 高风险动作审批

人工审批 / 人工接管必须上升为架构强约束：

- 读取类、低风险问答类场景，可自动执行
- 写数据、发消息、建工单、调用外部生产系统，必须二次确认或审批
- 财务、人事、合同、权限变更等高风险动作，必须人工审批或双人复核
- 当模型、检索、工具调用出现异常或低置信度时，必须可人工接管

规范要求：

- 默认拒绝，不默认放开
- `RBAC + ABAC` 组合使用
- 检索权限和工具权限不能共用一套粗粒度角色
- 审批链和授权链要有审计记录

当前方案对应组件：

- 身份源：企业现有 `SSO / IAM`
- 决策引擎：`Casbin`
- 执行点：`APISIX`、门户层、BFF、知识服务层、工具服务层、`MCP` 服务层
- 安全审查：`Qwen3Guard-Stream-8B`

### 10. 观测、评测与统一语义

企业级平台不能只有“有监控”，必须做到“统一语义的全链路观测”。

最低要求：

- 指标、日志、Trace 三者统一关联
- 用户请求、检索、重排、Prompt 组装、模型调用、工具调用、审批节点全部可追踪
- 质量评测、成本、时延、错误率、人工介入率都可统计

统一语义要求：

- 整个平台采用统一 `trace_id / session_id / app_id / user_id / tenant_id`
- AI 链路和系统链路可以在同一条追踪上串起来
- 模型调用、工具调用、检索调用应有标准属性字段
- 质量评测结果要能回链到具体版本和具体 Trace

推荐分工：

- `OpenTelemetry`：统一埋点、上下文传播、Trace 语义
- `Prometheus`：指标采集
- `Grafana`：看板、告警与分析视图
- `Loki`：日志聚合与检索
- `LangFuse`：LLM 链路追踪、Prompt 管理、质量评测、回放
- `k6`：发布前压测、容量基线与回归验证

规范要求：

- 日志、指标、Trace 不应各自孤立
- AI 评测数据和系统运维数据要能相互映射
- 评测不只做离线，还要有线上抽样和回放复盘

### 11. 异步作业层与事件化能力

即使当前不单列消息总线，异步作业能力也必须在架构上预留。

最低要求：

- 支持长时任务
- 支持批量索引
- 支持异步工具执行与回调
- 支持失败重试、幂等控制、状态追踪

典型作业包括：

- 文档同步与重建索引
- 大批量 Embedding 与向量重建
- 离线评测和回归测试
- 报告生成、文件导出、工单回调
- 跨系统长耗时任务执行

规范要求：

- 同步请求链路和异步作业链路必须分开
- 作业状态必须可追踪
- 幂等、重试、超时、死信处理要标准化

当前方案对应组件：

- 当前作为保留能力预留
- 早期可由数据库队列、Redis 队列或平台内置任务机制承载
- 规模上来后再演进到 `Kafka / Redpanda` 等消息总线

### 12. 基础设施、SRE 与平台运行规范

企业智能体平台最终仍然是生产系统，必须按生产系统治理。

最低要求：

- `dev / staging / prod` 三套环境
- 关键服务高可用
- 数据备份、恢复演练、容量规划
- GPU 资源池、模型池、数据区、观测区分离

规范要求：

- 平台容量按并发、知识规模、峰值任务量设计
- 模型池按能力拆池，避免通用模型、代码模型、安全模型抢占资源
- 门户、网关、编排、知识服务要支持水平扩容

当前方案对应组件：

- `APISIX`
- `K3s`
- `PostgreSQL`
- `Redis`
- `MinIO`
- 推理集群 `vLLM + Qwen`
- `k6`

## 五、企业生产基线清单

以下 13 项没有做到之前，不建议把平台定义为“企业生产级智能体平台”：

1. 有统一身份，不允许匿名绕过。
2. 有统一模型网关，不允许业务直接连底层模型。
3. 有正式数据治理层，不让 AI 应用直接连接原始业务系统。
4. 有正式知识链路，不是手工上传文件后直接问答。
5. 有正式编排模式，不靠一次性脚本拼装。
6. 有完整授权链路，不是只有一个角色表。
7. 有人工审批和人工接管强约束，不默认全自动执行高风险动作。
8. 有控制面，不在生产环境手工改 Prompt 和工具配置。
9. 有统一语义观测，能回放一次完整执行链路。
10. 有评测与发布门槛，先评测再灰度。
11. 有异步作业层，长时任务不阻塞前台请求链路。
12. 有多环境和高可用，不把试点环境直接当生产。
13. 有审计与数据治理，不让智能体天然拥有全库访问权。

## 六、与方案的关系

当前方案是“开源组件落地视角”的实施方案；本页是“能力与规范视角”的完整性框架。

两者的关系应当是：

- 本页定义企业级平台应具备什么
- 方案文档定义当前准备用哪些开源组件去承载这些能力
- 后续方案升级时，优先检查能力是否补齐，而不是先堆新组件

也就是说，未来方案升级的判断标准不应只是“多了几个产品”，而应是：

- 控制面是否更完整
- 授权链路是否更闭环
- 人工审批是否更刚性
- RAG 链路是否更真实
- 观测语义是否真正统一
- 异步作业能力是否真正进入正式底座

## 七、可参考的业界规范与标准方向

以下内容更适合作为企业能力体系的参考基线：

- 身份与访问：`OIDC / OAuth2 / SAML / SCIM`
- API 与工具描述：`OpenAPI`、`JSON Schema`
- Agent 互操作：`MCP`、`A2A`
- 数据治理方向：元数据目录、血缘、数据契约、分类分级、ACL 标签
- 观测与统一语义：`OpenTelemetry`
- 风险治理与 AI 管理：`NIST AI RMF`、企业内部合规规范
- 发布治理：版本化、灰度、回滚、变更审计、评测准入

这些规范不一定都对应单独组件，但它们决定了平台是不是完整、是不是可治理、是不是可长期演进。
