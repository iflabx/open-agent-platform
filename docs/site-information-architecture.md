# 站点信息架构

> 角色：文档站页面组织与导航规划基线
>
> 目标：明确一级页面、二级页面、详情页库之间的关系
>
> 说明：本页用于规划文档站的信息架构，不等于某一版技术方案本身。

## 一、页面组织原则

文档站建议采用三层结构：

1. 一级页面：回答整套方案最核心的 8 个主题。
2. 二级页面：展开一级主题下的主要子主题。
3. 详情页库：组件页、协议页、标准页、规范页、组件清单页，作为被引用的稳定说明页。

需要明确区分：

- 一级页面和二级页面属于“主题导航体系”
- 组件页、协议页、标准页、规范页属于“详情页库”
- 能力与规范页不要求一一对应到具体组件
- 组件、协议、标准、规范页只记录可落到方案某一层的实施项或治理基线

## 二、首页定位

首页不计入 8 个一级页面。

首页建议只承担四个职责：

- 说明站点主题
- 给出推荐阅读路径
- 跳转到方案
- 跳转到文档索引

## 三、8 个一级页面

### 1. 整体方案（能力和规范）

作用：

- 定义企业级智能体平台“应该具备什么”
- 从能力完整性和规范角度看整套体系

建议二级页面：

- 建设目标
- 设计原则
- 能力域总览
- 生产基线
- 业界规范与标准参考

当前主要对应：

- [企业智能体能力与规范体系](/capability-framework)

### 2. 方案

作用：

- 定义当前这一版到底采用什么方案
- 明确组件组合、职责边界、风险与取舍

建议二级页面：

- 方案范围
- 当前技术栈
- 关键取舍
- 组件职责边界
- 风险与限制
- 演进记录

当前主要对应：

- [方案](/framework)

### 3. 整体架构

作用：

- 从架构层面说明分层、链路、平面关系和部署关系

建议二级页面：

- 总体分层
- 核心业务链路
- 控制面 / 运行面 / 数据面 / 治理面
- 各层详细说明
- 部署分区与流量路径

各层详细说明建议继续拆为层级页：

- 模型服务层
- 模型网关层
- Agent 编排层
- 数据治理层
- 知识与检索层
- 门户与应用层
- 治理与观测层
- 基础设施层

当前主要对应：

- [总体架构](/architecture)

### 4. 数据治理与 AI 数据底座

作用：

- 单独讲企业结构化、非结构化数据如何采集、清洗、治理、管理，并供给给 AI 系统

建议二级页面：

- 数据源登记

- 非结构化知识治理
- 元数据与血缘
- 数据质量

- [数据治理层（企业 AI 数据底座）](/data-governance)

### 6. 开源状态与许可证

作用：

## 四、二级页面和详情页库的关系

二级页面不应该重复写组件细节。

更合理的分工是：

- 二级页面负责讲主题、边界、原则、分层关系和落地要求
- 组件页负责讲某个组件的定位、职责、适用场景和边界
- 协议页负责讲协议职责和互操作边界
- 标准页负责讲接口兼容、数据结构和统一语义标准
- 规范页负责讲授权模型、策略治理和人工审批要求
- 组件清单页负责讲“当前方案到底看哪些组件”

也就是说：

- 二级页面是“主题展开”
- 组件页、协议页、标准页、规范页是“被二级页面引用的详情页”

## 五、详情页库

详情页库建议长期保持稳定路径，不随公开导航频繁变化。

包括五类：

### 1. 组件页

例如：

- `vLLM`
- `LiteLLM`
- `AgentifUI`
- `LangChain`
- `LlamaIndex`
- `Letta`
- `PostgreSQL`
- `Redis`
- `MinIO`
- `Weaviate`

### 2. 协议页

例如：

- `MCP`
- `A2A`

### 3. 标准页

例如：

- `JSON Schema`
- `OpenAI 兼容 API`
- `OpenTelemetry`

### 4. 规范页

例如：

- `RBAC + ABAC`
- `Policy as Code`
- `Maker-Checker`

### 5. 组件清单页

例如：

- `components`

规则：

- 公开站点只展示当前方案与当前组件状态
- 组件页、协议页、标准页、规范页采用稳定路径复用
- 页面标题、导航名称和正文保持无编号名称

## 七、后续建设规则

后续如果继续扩站，建议遵守以下规则：

1. 先决定它属于哪个一级主题，再决定是不是需要单独页面。
2. 如果内容是“某个主题的展开”，优先做二级页面。
3. 如果内容是“某个具体实现项说明”，优先做组件页或协议页。
4. 如果内容只是当前方案选型结果，不要重复造组件页，放进组件清单页即可。
5. 只有当页面能承载稳定复用价值时，才值得进入长期导航体系。

## 八、建议的后续实际导航重构顺序

不建议一次性把站点导航全部改完。

更稳妥的顺序是：

1. 先按本页确定一级页面结构。
2. 再为 8 个一级页面逐步补齐对应主文档。
3. 然后再决定哪些一级页面需要真正拆成二级页面。
4. 最后才重构实际站点导航和侧边栏。

这样能避免信息架构还没稳定，页面路径就反复调整。
