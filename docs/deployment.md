# 部署与发布

## 文档站发布到 GitHub Pages

本项目已经内置 `.github/workflows/deploy.yml`，采用 GitHub Actions 自动发布。

### 使用方式

1. 将当前目录初始化为 Git 仓库并推送到 GitHub。
2. 默认分支使用 `main`。
3. 在 GitHub 仓库设置中启用 `Pages`，来源选择 `GitHub Actions`。
4. 每次向 `main` 分支推送后，工作流会自动构建并发布文档站。

### 关键实现点

- 构建工具为 `VitePress`
- 构建命令为 `npm run docs:build`
- 发布目录为 `docs/.vitepress/dist`
- `base` 路径由 `GITHUB_REPOSITORY` 自动推导，适配仓库名路径前缀

## 平台本体部署建议

如果后续不仅发布文档站，还要落地智能体平台本身，建议采用分阶段部署策略。

## 阶段一：单团队试点

推荐部署：

- 1 套应用服务
- 1 套 PostgreSQL
- 1 套 Redis
- 1 套模型服务节点
- 对象存储可先复用现有平台

适合团队在低成本条件下完成知识助手或服务台场景验证。

## 阶段二：部门级推广

推荐增加：

- 独立入口网关 `APISIX`
- Keycloak 统一身份认证
- 独立的日志、指标和追踪系统
- 检索层与业务层分离部署

这个阶段的重点不是“功能更多”，而是稳定性、权限隔离和运维标准化。

## 阶段三：企业级生产

推荐能力：

- Kubernetes 多环境部署
- GPU 推理集群与弹性伸缩
- 向量检索独立集群
- 多租户隔离策略
- 灰度发布、容量规划和灾备方案
- 基于 `k6` 的发布前容量基线与回归压测

## 压测与容量验证

建议把 `k6` 作为统一压测工具，覆盖以下链路：

- `APISIX` 入口网关
- 门户 `BFF / API`
- 知识检索与重排服务
- Agent 服务与工具调用入口
- `LiteLLM` 模型网关接口

## 运维建议

### 最少要监控的指标

- 请求量、成功率、P95 时延
- 入口网关上游错误率和限流命中率
- 模型 Token 消耗和单位成本
- 工具调用成功率和平均耗时
- 检索命中率与引用命中率
- 安全策略触发次数和人工接管比率

### 推荐的环境划分

- `dev`：面向研发联调
- `staging`：面向真实数据的预发布验证
- `prod`：面向业务生产流量

不要把 Prompt 调整、知识索引更新和工具接入直接在生产环境手工修改，而应走标准变更流程。
