# 文档编写与维护指南

> 角色：仓库级文档编写与维护基线
>
> 适用范围：当前文档站所有公开页面与后续新增文档
>
> 目标：统一页面类型、命名方式、写作结构、更新流程和联动范围，避免站点结构失控
>
> 说明：本页约束的是“怎么写、怎么改、改了以后要同步什么”，不替代具体方案页、组件页、协议页、标准页、规范页的业务内容

相关文档：

- [文档索引](/doc-index)
- [方案](/framework)
- [组件](/components)
- [开源状态与许可证说明](/open-source-status)

## 一、先明确这个站点是什么

这个仓库首先是一个公开文档站，其次才是一个用于持续维护方案内容的仓库。

因此，后续写文档时要先守住四条总规则：

1. 公开站点始终只展示一个当前状态，不展示内部方案版本号。
2. 公开一级入口固定使用无版本名称，例如 `方案`、`组件`、`开源状态`。
3. 主方案页固定发布到 `/framework`，组件清单页固定发布到 `/components`。
4. 组件页、协议页、标准页、规范页采用稳定路径复用，不新增公开的版本化页面。

站点公开标题也保持统一：

- 首页主标题使用 `企业级智能体开源系统方案`
- 站点标题使用 `企业级智能体开源系统方案`
- 相应公开标题保持同一口径，不出现内部迭代标识

## 二、页面类型怎么区分

在当前站点里，文档大致分成四类，写作目的不同，不能混写。

### 1. 首页与总索引

- 首页负责说明站点主题、给出推荐阅读路径、把读者引到主方案或总索引。
- 总索引负责汇总所有主题页和详情页入口，属于导航页，不承担细节论证。

对应页面：

- [首页](/)
- [文档索引](/doc-index)

### 2. 主题页

主题页回答的是“这一类问题整体怎么设计”，重点是范围、原则、边界、链路和落地要求。

当前主题页包括：

- [方案](/framework)
- [企业智能体能力与规范体系](/capability-framework)
- [协议与标准体系](/protocol-standards)
- [数据治理层（企业 AI 数据底座）](/data-governance)
- [总体架构](/architecture)
- [扩展能力与补齐方向](/extension-capabilities)
- [技术选型](/stack)
- [典型场景](/scenarios)
- [安全与治理](/governance)
- [部署与发布](/deployment)
- [实施路线图](/roadmap)

### 3. 详情页库

详情页回答的是“具体这个对象是什么，在方案里承担什么职责，有什么边界”。

详情页库包括四种：

- 组件页：例如 `/components/litellm`
- 协议页：例如 `/protocols/mcp`
- 标准页：例如 `/standards/opentelemetry`
- 规范页：例如 `/norms/policy-as-code`

这类页面应长期保持稳定路径，优先修改原页面，而不是新增公开替代页。

### 4. 维护支撑页

维护支撑页服务于写作和维护过程，不负责承担方案本体内容。

当前维护支撑页包括：

- [文档编写与维护指南](/documentation-maintenance)
- [第一批文档完善计划](/first-batch-documentation-plan)
- [第二批文档完善计划](/second-batch-documentation-plan)

## 三、不同页面分别该写什么

### 1. 主题页

主题页应该优先回答以下问题：

- 为什么要单独讨论这个主题
- 这个主题在整套方案里的位置是什么
- 设计原则和边界是什么
- 当前方案如何落位
- 与其他主题页、详情页是什么关系

主题页不要把内容退化成“组件堆名单”。如果只是列产品名、协议名、许可证名，那就应当把细节放回详情页库。

### 2. 组件页

组件页建议保持当前已形成的稳定结构：

- `类型`
- `复用规则`
- `所属层`
- `当前定位`
- `主要职责`
- `与其他组件关系`
- `适合场景`
- `边界`
- `采用规则`
- `治理注意点`
- `关联文档`
- `参考资料`

组件页重点是“职责和边界”，不是产品百科，也不是部署手册。

推荐理解方式：

- `类型` 用来说明它在方案中的角色，而不是官方产品分类。
- `复用规则` 用来说明它是稳定复用、条件引入、互斥候选还是配套组件。
- `所属层` 只写一个主归属层，避免同一组件在多个层重复解释。
- `当前定位` 用一段话说明“为什么当前方案要用它”。
- `主要职责` 只列当前方案真正依赖的能力，不铺产品全量特性。
- `与其他组件关系` 重点写上下游关系、依赖关系和不可替代关系。
- `适合场景` 回答“什么情况下应该使用它”。
- `边界` 回答“什么不该让它做、它不能替代什么”。
- `采用规则` 回答“默认采用、条件引入还是互斥使用”。
- `治理注意点` 只写和当前方案落地直接相关的权限、审计、数据和发布要求。
- `关联文档` 用来连接层文档、主题页和相关组件页。
- `参考资料` 统一放在文档底部，优先列官方文档、官方仓库和官方许可证页面。

组件页可直接套用以下模板：

```md
# 组件名称

> 类型：一句话说明组件类型
>
> 复用规则：稳定复用 / 条件引入 / 互斥候选 / 配套组件
>
> 所属层：九层中的主归属层名称

## 当前定位

用一段话说明它在当前方案中的角色。

## 主要职责

- 职责 1
- 职责 2
- 职责 3

## 与其他组件关系

- 与上游组件的关系
- 与下游组件的关系
- 不能替代的组件或能力

## 适合场景

- 场景 1
- 场景 2

## 边界

- 不负责什么
- 不替代什么
- 不建议怎么使用

## 采用规则

- 默认采用还是条件引入
- 是否存在互斥对象
- 何时需要升级或替换

## 治理注意点

- 权限或审计要求
- 数据边界要求
- 发布和运维注意点

## 关联文档

- [所属层文档](/layers/example)
- [相关主题页](/framework)
- [相关组件页](/components/example)

## 参考资料

- [官方文档](https://example.com)
- [官方仓库](https://github.com/example/example)
```

### 3. 协议页

协议页重点回答：

- 这个协议解决什么互操作问题
- 在当前方案里放在哪一层
- 适用场景是什么
- 不应该拿它替代什么

例如 `MCP` 与 `A2A` 必须明确分工，避免协议职责混淆。

### 4. 标准页

标准页重点回答：

- 这个标准约束什么
- 是接口兼容、数据结构，还是运行语义
- 在门户、工具、模型、观测或治理链路中的落点是什么

### 5. 规范页

规范页重点回答：

- 企业级平台必须遵守什么治理基线
- 哪些能力是强约束，哪些是建议项
- 哪些环节需要审批、审计、版本化和回滚

## 四、命名、标题和路由规则

### 1. 标题规则

- 面向公开站点的页面标题不出现内部版本号或内部迭代标识
- 页面标题优先使用稳定中文名称
- 组件、协议、标准、规范名称保持行业常见叫法，不随内部讨论频繁改名

### 2. 路由规则

- 公开主方案页固定为 `/framework`
- 组件清单页固定为 `/components`
- 组件详情页、协议页、标准页、规范页使用稳定英文或约定俗成 slug
- 已存在稳定路径时，优先复用，不新增“新版页”“临时页”“替代页”

### 3. 文件命名规则

- 文档文件名使用小写英文和短横线
- 不在文件名里写临时阶段标记，例如 `draft`、`final-v2`
- 如果对象已经存在正式详情页，直接改原文件

## 五、写作风格规则

### 1. 先写边界，再写能力

对方案类文档，最容易失控的地方不是信息不够，而是边界不清。写作时优先回答：

- 这页负责解释什么
- 不负责解释什么
- 哪些细节应跳转到其他页

### 2. 先写当前方案，再写备选和演进

公开站点优先表达“当前推荐状态”，不要把多个历史版本同时铺在正文里。需要保留备选路线时，应明确：

- 哪些是当前方案
- 哪些是互斥候选
- 哪些是后续演进方向

### 3. 区分三种表述

- `必须`：架构强约束或治理基线
- `建议`：当前推荐做法
- `可选`：条件成立时再引入的能力

这三种语气不要混用，否则文档会既像原则，又像随手建议。

### 4. 组件、协议、标准名称建议加反引号

例如：

- `LiteLLM`
- `MCP`
- `OpenTelemetry`

这样更容易把“专有名词”和正文叙述区分开。

### 5. 少写宣传语，多写可执行判断

优先写这些内容：

- 适用场景
- 主要职责
- 与上下游关系
- 边界和限制
- 采用条件
- 风险与注意事项

少写空泛表述，例如“功能强大”“生态完善”“体验优秀”。

### 6. 参考资料统一放在文档底部

凡是编写过程中引用了权威资料、官方文档、规范正文、官方仓库或官方许可证页面，都应在文档底部单独设置 `参考资料` 一节，明确列出参考链接。

执行要求：

- 优先引用官方一手资料，不用二手转述替代原始来源。
- `参考资料` 放在正文末尾，便于读者统一查看。
- 如果正文里基于某个关键规范或官方口径做了重要判断，底部必须能对应找到来源链接。
- 即使正文没有逐段脚注，只要该页参考了外部权威资料，也要在底部集中列出。

## 六、常见联动关系

一个文档变了，通常不止改一页。下面这些联动最容易漏。

### 1. 主方案变化

如果 [方案](/framework) 中的当前组件组合、主平台路线或层次边界发生变化，通常至少要同步：

- [组件](/components)
- [技术选型](/stack)
- [总体架构](/architecture)
- [开源状态与许可证说明](/open-source-status)

### 2. 能力基线变化

如果 [企业智能体能力与规范体系](/capability-framework) 的能力域或治理要求发生变化，通常要检查：

- [协议与标准体系](/protocol-standards)
- [安全与治理](/governance)
- [数据治理层（企业 AI 数据底座）](/data-governance)
- 对应的规范页和标准页

### 3. 协议或标准变化

如果协议层、接口标准或观测语义变化，通常要同步：

- [协议与标准体系](/protocol-standards)
- 对应协议页、标准页
- [方案](/framework)
- [总体架构](/architecture)

### 4. 组件清单变化

如果新增或移除正式组件，通常要同步：

- [组件](/components)
- 对应详情页
- [文档索引](/doc-index)
- `docs/.vitepress/config.mts` 中的侧边栏
- 必要时同步 [开源状态与许可证说明](/open-source-status)

### 5. 路由或页面结构变化

如果新增主题页或调整页面结构，通常要同步：

- [文档索引](/doc-index)
- `docs/.vitepress/config.mts` 中的导航与侧边栏

## 七、新增页面时的最小检查清单

### 1. 新增主题页

- 是否真的需要新增主题，而不是补充到现有主题页
- 页面职责是否清楚
- 是否已经在 [文档索引](/doc-index) 中加入入口
- 是否需要加入侧边栏
- 是否与内部结构台账保持一致

### 2. 新增组件页

- 是否已经确认它属于正式清单，而不是研究候选
- 页面是否写清 `当前定位`、`职责`、`边界`
- 是否更新 [组件](/components)
- 是否更新 [文档索引](/doc-index)
- 是否需要补充 [开源状态与许可证说明](/open-source-status)

### 3. 新增协议页、标准页、规范页

- 是否属于稳定复用的详情页，而不是临时分析页
- 是否写清适用场景与边界
- 是否更新对应总览页
- 是否在 [协议与标准体系](/protocol-standards) 或相关主题页中补足落点

## 八、修改文档前后的自检

修改前先问三件事：

1. 这次改动属于主题页、详情页，还是维护支撑页
2. 我是在更新当前方案，还是在追加历史讨论
3. 哪几页会因此产生不一致

修改后至少检查以下项目：

- 页面标题是否符合公开站点规则
- 是否出现内部版本号或临时标记
- 是否把“能力基线”和“具体组件”混写了
- 是否把“当前方案”和“候选项”混写了
- 若参考了外部权威资料，文档底部是否已补 `参考资料` 和对应链接
- 相关页面是否已同步
- 新增页面是否已补入口

## 九、推荐写作顺序

如果要写一个全新主题，推荐按下面顺序推进：

1. 先在内部结构台账里确认它属于哪类页面
2. 先写主题页的范围、目标、边界和与其他页面关系
3. 再补对应详情页库
4. 最后更新 [文档索引](/doc-index) 和侧边栏

如果要写一个新的具体对象，推荐按下面顺序推进：

1. 先判断它是组件、协议、标准还是规范
2. 先补详情页
3. 再补到对应总览页和索引页
4. 最后检查是否要影响主方案、架构或开源状态说明

## 十、一句话原则

这套文档站优先表达“当前公开方案是什么、为什么这样设计、边界在哪里、落地时要注意什么”。

凡是会破坏这一目标的写法，都应收敛：

- 不堆历史版本
- 不堆无边界的候选项
- 不把主题页写成清单页
- 不把详情页写成百科页
- 不让一个变更只改一处而导致全站失真
