# k6

> 类型：压测与容量验证
>
> 复用规则：稳定复用的组件说明页
>
> 所属层：治理与观测层

## 当前定位

`k6` 用于对当前方案中的入口网关、门户 BFF、知识服务、Agent 服务和模型接口做性能基线、压力测试与容量验证。

## 主要职责

- HTTP / API 压测
- 关键链路基线测试
- 峰值并发与瓶颈识别
- 发布前容量验证

## 与其他组件关系

- 可对 [APISIX](/components/apisix)、门户层、编排层、知识服务和模型接口执行统一压测。
- 与 [Prometheus](/components/prometheus)、[Grafana](/components/grafana) 等观测体系配合分析容量瓶颈。
- 与 [LangFuse](/components/langfuse) 互补，前者看性能容量，后者看 LLM 质量链路。
- 不直接参与生产流量，而是服务于发布前验证和容量评估。

## 适合场景

- 入口网关压测
- 门户与 BFF 性能验证
- 知识检索与模型接口基线验证

## 边界

- 不替代 `LangFuse` 的 LLM Trace 与评测
- 不替代长期在线监控体系
- 不直接参与生产请求链路

## 采用规则

- 当前方案默认以 `k6` 作为关键链路的压测与容量验证工具。
- 重要发布、架构调整或容量变化前，应对入口、检索、编排和模型接口做基线校验。
- 压测结果应进入发布门槛，而不是停留在临时测试报告中。

## 治理注意点

- 压测场景、脚本、目标环境和结果基线需要版本化保存。
- 压测要区分开发、预发和生产保护边界，避免误伤真实业务。
- 容量结果应和指标、错误率、时延及业务成功率一起解读。
- 仓库许可证口径需与 [开源状态与许可证说明](/open-source-status) 保持一致。
