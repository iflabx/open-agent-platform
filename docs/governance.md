# 安全与治理

## 治理目标

企业级智能体系统必须兼顾四件事：

- 正确的人访问正确的数据
- 每一步操作都能被追踪和审计
- 模型输出受到策略约束
- 平台变更具备可验证的质量门槛

## 身份与权限

### 统一身份认证

建议通过企业 SSO 接入统一身份源，并将用户、部门、角色和租户信息注入智能体运行时。这样可以把“谁可以看到什么”“谁可以执行什么”前移到平台层，而不是散落在各个应用里。

入口链路建议统一收敛为：`SSO / IAM -> APISIX -> BFF / Portal -> Casbin`。这样认证、入口审计、流量治理和细粒度授权能形成完整链路。

### 权限控制模型

常见做法是组合使用：

- `RBAC` 处理基础角色授权
- `ABAC` 处理部门、项目、数据级标签等细粒度约束
- 工具级授权处理高风险动作，例如发送通知、写入 CRM、执行 SQL

## 数据安全

### 关键控制点

- 文档入库前做分类分级、脱敏和标签治理
- 检索时按用户上下文做权限过滤
- Prompt 拼装时避免把越权数据注入模型上下文
- 输出前做敏感词、PII、合规策略和模板规则检查

### 数据最小化原则

智能体不应天然拥有“全库访问权”。任何检索范围、工具调用范围和历史上下文保留策略都应遵循最小权限原则。

## 审计与可观测

建议至少记录以下链路：

| 维度 | 关键字段 |
| --- | --- |
| 用户访问 | 用户 ID、角色、租户、入口渠道、会话 ID、网关路由 / 应用 ID |
| 检索过程 | 数据源、命中条目、权限过滤结果、重排得分 |
| 模型调用 | 模型标识、输入摘要、输出摘要、Token 成本、时延 |
| 工具执行 | 工具名称、参数摘要、调用结果、失败原因、回滚动作 |
| 安全策略 | 命中的规则、审查结论、人工接管标记 |

## 评测与发布门槛

### 评测维度

- 问答正确率
- 引用有效性
- 工具调用成功率
- 高风险动作误触发率
- 平均响应时延和成本
- 发布前容量基线是否达标

### 发布机制

建议把 Prompt、工具、知识索引配置和模型路由都纳入版本管理，任何影响行为的变更都应经过：

1. 离线评测
2. `k6` 压测与容量基线验证
3. 小流量灰度
4. 审计抽样复盘
5. 正式发布

## 人工兜底

只要系统开始涉及审批、数据写入、外部通知、支付或合同类流程，就必须在设计上保留人工确认和人工接管节点。企业级系统的目标不是完全无人，而是可靠地减少重复劳动并控制风险。
