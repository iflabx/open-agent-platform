# 总体架构

## 架构总览

企业级智能体系统建议采用六层结构：接入与交互层、智能编排层、数据治理层、知识与数据服务层、治理与观测层、基础设施层。这样可以同时兼顾业务交付速度和平台稳定性。

如需单独查看“企业 AI 数据底座”设计，可参考 [数据治理层（企业 AI 数据底座）](/data-governance)。

## 分层说明

### 1. 接入与交互层

面向最终用户和外部系统的入口，包括：

- 统一入口网关 `APISIX`
- Web 门户、管理后台、内部工作台
- 企业 IM、邮件、工单系统、CRM、ERP 等渠道集成
- 面向其他系统开放的 REST API / Webhook / SDK

这一层的重点不是复杂业务逻辑，而是统一入口、统一流量治理、统一身份和统一体验。

### 2. 智能编排层

这是平台的大脑，负责：

- 多 Agent 任务拆解与状态管理
- Prompt 模板管理与上下文编排
- 工具调用路由、回退、重试和人工确认
- 工作流执行、事件触发和跨系统编排

如果平台未来要支持多个业务线，这一层必须采用平台化设计，而不是写死在单个应用里。

### 3. 数据治理层

这一层位于原始业务系统与 AI 消费层之间，负责把原始数据治理成可供 AI 使用的数据资产和数据服务：

- 结构化与非结构化数据采集
- 清洗、标准化、去重和格式治理
- 元数据、血缘、标签和生命周期管理
- 权限标签和分类分级注入
- 向 `RAG / 搜索 / 查询 / Agent` 发布治理后的数据服务

设计关键点是统一接入、责任人、数据质量、权限标签和数据产品发布，而不是把原始系统直接暴露给智能体。

### 4. 知识与数据服务层

该层负责支持智能体的“长期记忆”和“受控访问”：

- 企业文档、制度、FAQ、产品资料、工单记录
- Embedding、向量检索、重排和结构化查询
- 会话历史、任务状态、操作日志和评测结果

设计关键点是权限过滤、索引更新、引用追踪和基于治理后数据资产的受控访问。

### 5. 治理与观测层

企业级方案与 Demo 的差别，主要体现在这一层：

- SSO、RBAC、ABAC、租户隔离
- 审计日志、链路追踪、提示词与输出留痕
- 观测指标、告警规则、成本统计、质量评测
- 发布前压测与容量验证
- 输出安全策略、脱敏、内容审核和人工兜底

### 6. 基础设施层

支撑平台运行的基础资源，包括：

- Kubernetes、容器运行时、CI/CD
- GPU 推理集群、对象存储、消息队列
- PostgreSQL、Redis、向量数据库、日志系统
- `APISIX`、服务网格、证书与密钥管理

## 核心链路

一个典型请求建议经过以下流程：

1. 用户通过门户或企业 IM 发起问题或任务。
2. `APISIX` 执行入口级路由、限流、审计和灰度治理，并协同认证接入。
3. 接入层完成租户识别、会话初始化和上下文传递。
4. 编排层根据意图决定使用问答模式、工作流模式或多 Agent 协作模式。
5. 数据治理层持续把企业原始数据治理成可供 AI 使用的数据资产和数据服务。
6. 知识服务层执行检索、过滤和重排，并返回受权限约束的上下文。
7. 编排层调用模型服务、工具服务和外部业务系统。
8. 治理层记录执行轨迹，执行安全策略、质量检查和容量基线验证。
9. 最终结果回到用户界面，同时留存审计和评测数据。

## 推荐模块边界

| 模块 | 职责 | 边界建议 |
| --- | --- | --- |
| 模型网关 | 统一适配不同推理后端 | 不承载业务逻辑 |
| Ingress Gateway | 统一北向入口、路由、限流、灰度和审计 | 不替代模型网关与授权决策 |
| Agent Runtime | 管理状态、任务、工具调用 | 不直接处理权限策略 |
| Data Governance Service | 数据采集、清洗、标准化、元数据与数据产品发布 | 不直接持有 UI 交互逻辑 |
| Knowledge Service | 文档同步、切片、索引、检索 | 不直接耦合 UI |
| Tool Hub | 封装 ERP、CRM、工单、数据库工具 | 所有高风险操作都要求审计 |
| Governance Service | 权限、审计、脱敏、评测 | 对外提供统一策略接口 |
| Portal & Admin | 用户入口和运维后台 | 不持有核心智能逻辑 |
