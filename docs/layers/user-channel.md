# 1. 用户与渠道层

> 定位：承接员工、运营、业务系统和外部渠道进入智能体平台的所有前门。
>
> 当前承载对象：企业现有 `Web / IM / 工单 / 邮件 / CRM / ERP / Webhook` 入口。

- [方案](/framework)

## 一、本层定义

用户与渠道层是企业智能体系统最靠近真实使用者的一层，负责把“人”和“外部业务事件”稳定送入平台。

这一层的重点不是新增一个统一前端组件，而是复用企业已经存在的入口体系，把访问入口、会话入口、消息入口和业务触发入口纳入统一治理链路。

## 二、本层功能

- 承接员工门户、企业 IM、工单系统、邮件和业务系统入口。
- 提供聊天、表单、工单发起、消息通知、Webhook 回调等进入方式。
- 把用户身份、组织、渠道、来源系统等上下文传给下游入口层和应用层。
- 负责把用户反馈、人工转办、会话继续等动作回传到对应业务入口。

## 三、当前承载对象

当前方案在这一层不新增正式开源组件，主要复用企业已有能力：

- 企业 Web 门户
- 企业 IM
- 工单系统
- 邮件入口
- CRM / ERP / 业务系统入口
- Webhook / 外部系统回调入口

这意味着本层的设计重点是“统一接入和统一治理”，不是“重新造一个渠道平台”。

## 四、上下游关系

上游对象：

- 员工、运营、客服、管理人员
- 企业内部业务系统
- 外部系统事件和回调

下游层级：

- [统一接入与流量治理层](/layers/access-traffic-governance)
- [门户与应用层](/layers/portal-application)

默认链路应保持为：

`渠道入口 -> 统一接入 -> 门户 / BFF -> Agent 编排`

## 五、边界

本层负责“进入平台”，但不负责以下能力：

- 不直接调用模型或维护模型路由。
- 不直接访问知识索引或原始业务数据。
- 不承担细粒度授权裁决。
- 不承载工作流编排和工具调度逻辑。

如果渠道入口直接内嵌这些能力，平台就会重新回到“多入口各做一套”的失控状态。

## 六、关键链路

### 1. 人机交互链路

1. 用户从 Web、IM 或工单入口进入。
2. 渠道入口把身份和来源上下文交给统一接入层。
3. 统一接入层完成认证接入、路由和前置治理。
4. 门户 / 应用层或 Agent 编排层继续处理业务逻辑。

### 2. 事件触发链路

1. CRM / ERP / 工单系统通过 API 或 Webhook 发起事件。
2. 事件先经过统一接入层。
3. 下游平台根据规则触发应用、流程或通知。

## 七、治理要求

- 同一用户在不同渠道中的身份映射必须可追踪。
- 渠道入口必须保留来源系统、会话标识和时间信息，便于审计。
- 用户可见界面应满足基本可访问性要求，避免把关键能力限制在单一交互模式中。
- 渠道层不应绕过统一入口直接访问编排层、知识层或模型层。

## 八、部署与发布要求

- 优先复用企业现有门户、IM 和业务系统入口，不额外引入新的前门。
- 入口流量统一收敛到网关层，避免每个渠道单独暴露后端服务。
- 渠道发布节奏可以快于后端平台，但接口契约要通过统一 API 管理。
- 渠道侧只保留最薄的一层适配逻辑，业务流程和智能能力放到下游层处理。

## 九、风险与取舍

- 如果每个渠道都各自直连后端，身份、审计和限流会快速失控。
- 如果把过多业务逻辑写在渠道适配层，后续迁移和复用成本会很高。
- 如果只关注聊天入口，忽略工单、邮件和业务系统触发，平台很难真正进入企业流程。
