# 数据治理层（企业 AI 数据底座）

> 角色：企业结构化与非结构化数据面向 AI 的统一治理与供给层
>
> 目标：把企业原始数据变成可治理、可授权、可复用、可追溯的 AI 可消费数据服务
>
> 说明：这不是单一“向量库”或“知识库”组件，而是一整条从采集、清洗、治理到供给给 `RAG / 搜索 / 查询 / Agent` 的平台能力链。

## 一、为什么要单独有一个数据治理层

企业 AI 平台最大的问题之一，是数据常常直接从业务系统“临时拉取”给智能体使用。

这样做的问题很明显：

- 数据源分散，质量不稳定
- 权限模型不一致
- 结构化和非结构化数据各自孤立
- 没有统一元数据、责任人和生命周期管理
- 每个 AI 应用各自接数据，重复建设且不可复用

因此，企业智能体方案里应单独存在一个“数据治理层”，它位于原始业务系统与 AI 应用之间，负责把原始数据治理成标准化的数据资产和 AI 数据服务。

## 二、这层的定位

可以把数据治理层理解为：

`源系统 -> 数据治理层 -> AI 数据服务 -> RAG / 搜索 / 查询 / Agent`

它不是替代业务系统，也不是替代知识库，而是统一承接以下工作：

- 数据采集
- 数据标准化
- 数据清洗
- 元数据治理
- 权限与标签治理
- 数据资产目录
- AI 可消费接口发布

## 三、治理范围

### 1. 结构化数据

- MySQL / PostgreSQL / Oracle / SQL Server
- 数仓、湖仓、报表系统
- ERP、CRM、HR、财务、供应链系统
- 工单、订单、客户、产品、组织、项目等业务对象

### 2. 非结构化数据

- PDF、Word、Excel、PPT
- Wiki、知识库、制度文档、产品说明
- 邮件、会议纪要、FAQ、表单附件
- 代码仓库、接口文档、运维文档、日志文档

### 3. 半结构化和多模态数据

- JSON、CSV、XML
- 图片、扫描件、截图
- 音视频转写文本

## 四、标准链路

### 1. 数据源登记

每个数据源都应先登记，而不是直接接入。

登记信息至少包括：

- 数据源名称
- 系统 owner
- 数据责任人
- 接入方式
- 更新频率
- 数据分类分级
- 是否含敏感数据
- 可供哪些 AI 场景使用

### 2. 数据采集

采集方式通常包括：

- 批量同步
- 增量同步
- CDC
- API 拉取
- 文件投递
- 事件驱动

关键要求：

- 采集过程要可追踪
- 失败要可重试
- 同步频率要可配置
- 不同源系统要有统一接入规范

### 3. 原始落地区

不建议采集后直接进入 AI 索引。

应至少区分三类区域：

- Raw：原始数据，保留原貌
- Staged：清洗和标准化中间层
- Curated：可供 AI 使用的治理后数据层

这样做的好处是：

- 可回溯
- 可重建
- 可比较治理前后差异
- 便于审计和质量追踪

### 4. 解析、清洗与标准化

这是把“数据”变成“可用数据资产”的关键步骤。

典型处理包括：

- OCR 和版面解析
- 文本抽取
- 去重与去噪
- 编码和格式标准化
- 字段标准化
- 实体抽取
- 时间、组织、产品、人员、项目等主数据对齐

结构化数据和非结构化数据都应被统一加上可治理的元数据，而不是只保留原始内容。

### 5. 元数据、血缘与资产目录

企业 AI 数据底座必须有统一元数据。

至少应治理这些信息：

- 数据来源
- 更新时间
- 数据 owner
- 数据责任部门
- 权限标签
- 业务标签
- 生命周期状态
- 版本号
- 血缘关系

这一步的意义在于：

- 知道一段知识从哪里来
- 知道为什么能被某个用户看到
- 知道什么时候应该失效、重建或下线

### 6. 权限标签与合规治理

AI 平台不能只在检索时“临时判断权限”，权限应在数据治理阶段就被写进资产。

最低要求：

- 数据分级分类
- 角色/部门/项目标签
- 敏感字段标识
- 数据域和租户边界
- 保留期与销毁策略

对 AI 来说，最关键的是：

- 权限标签能进入 RAG 索引
- 权限标签能进入搜索索引
- 权限标签能进入查询层

这样不同 AI 系统才能共用同一套数据资产，而不是各自重复做权限过滤。

### 7. 数据质量规则

如果没有质量规则，AI 最终只会放大脏数据问题。

常见规则包括：

- 唯一性
- 完整性
- 一致性
- 时效性
- 准确性
- 去重率
- 解析成功率

数据质量结果应当可量化，并作为索引入库和应用发布的前置条件之一。

### 8. AI 可消费数据服务发布

数据治理层的目标不是“存起来”，而是“发布为可消费服务”。

面向 AI，通常需要发布四类能力：

1. `RAG` 数据服务
   - 为知识问答和智能助手提供治理后的文档与片段
2. 搜索数据服务
   - 为全文搜索、混合搜索、应用内搜索提供索引服务
3. 查询数据服务
   - 为结构化数据问答、报表、指标查询提供受控查询接口
4. 语义上下文服务
   - 为 Agent 提供实体、关系、业务定义、主数据和上下文标签

关键点是：AI 系统消费的是“服务化后的治理数据”，而不是直接连接各类原始系统。

### 9. 生命周期管理

数据治理层必须负责数据资产的全生命周期：

- 新增
- 更新
- 失效
- 归档
- 重建索引
- 下线

否则 AI 系统很容易出现“旧数据还在索引里”“权限变了但索引没更新”“文档删除了但回答还引用”的问题。

## 五、给 AI 系统的供给方式

### 1. 给 RAG

- 供给治理后的文档片段
- 供给来源元数据
- 供给 ACL 标签
- 供给 Embedding / Rerank 所需字段

### 2. 给搜索

- 供给全文索引
- 供给关键词、标签、结构化过滤条件
- 供给混合检索所需向量与元数据

### 3. 给查询型 Agent

- 供给结构化数据查询视图
- 供给指标口径、字段释义和业务语义层
- 供给可审计的受控查询接口

### 4. 给多 Agent 系统

- 供给统一实体与主数据
- 供给跨系统上下文
- 供给可信来源引用

## 六、建议的模块拆分

一个企业级数据治理层，建议拆成以下几个模块：

| 模块 | 主要职责 | 输出给谁 |
| --- | --- | --- |
| Source Registry | 数据源登记、owner、分类分级、同步配置 | 平台管理层 |
| Ingestion Service | 批量、增量、CDC、API、文件采集 | 原始落地区 |
| Processing Pipeline | 解析、OCR、清洗、标准化、实体抽取 | 治理后数据层 |
| Metadata Catalog | 元数据、标签、血缘、责任人、版本 | 所有 AI 系统 |
| Policy Tagging | ACL、数据域、敏感级别、合规标签 | 检索层、查询层 |
| Indexing Service | 向量索引、全文索引、混合索引构建 | `RAG / 搜索` |
| Query Service | 结构化查询与语义查询服务 | 查询型 Agent |
| Data Product API | 发布标准数据服务与上下文服务 | 各类 AI 应用 |

## 七、推荐开源组合

针对当前方案目标，即同时满足 `RAG / 搜索 / 查询 / Agent` 的数据需求，又不把数据治理层做得过重，当前方案推荐采用以下轻中型组合：

- `OpenMetadata`
- `SeaTunnel`
- `dbt Core`
- `Apache Tika`

角色分工如下：

- `OpenMetadata`：统一元数据治理、资产目录、责任人、标签和血缘中枢
- `SeaTunnel`：统一采集、同步、批量、增量和 CDC 接入
- `dbt Core`：结构化数据标准化、建模、口径统一和治理后数据层产出
- `Apache Tika`：非结构化文档文本与元数据抽取

这组组件的目标不是替代 `RAG` 平台或向量库，而是先把企业原始数据治理成可供 AI 使用的数据资产和数据服务。

## 八、与方案的对应关系

当前方案其实已经覆盖了这层的一部分，但还不完整。

已能承载的部分：

- 原始文件与对象存储：`MinIO`
- 元数据与平台配置：`PostgreSQL`
- 运行时缓存与任务状态：`Redis`
- 知识工程和索引编排：`LlamaIndex`
- 平台内知识接入：`Dify`、`RAGFlow`、`Coze Studio`
- 向量与检索底座：`Weaviate`、`Milvus`、`Elasticsearch`

当前仍缺失或需要强化的部分：

- 统一数据源登记
- 统一元数据目录
- 血缘和生命周期治理
- 数据质量规则体系
- 结构化与非结构化统一的数据资产模型
- 面向 AI 的统一数据产品发布方式

也就是说，当前方案现在更偏“知识与索引消费层”，还需要补成完整“数据治理与数据供给层”。

## 九、可选增强组件方向

这一层不一定在当前阶段马上定死组件，但如果后续要落更完整的数据治理体系，常见会补这些类型的能力：

- 数据采集 / 同步：`Airbyte`、`Apache NiFi`、`SeaTunnel`
- 元数据目录 / 血缘：`OpenMetadata`、`DataHub`
- 数据转换 / 质量：`dbt`、`Spark`、规则化质量校验流程
- 联邦查询 / 语义查询：`Trino`、受控 SQL 服务

这些不是当前方案的强绑定清单，但它们属于“数据治理层”在企业级落地时常见的补全方向。

## 十、最关键的原则

这层真正要解决的，不是“数据有没有接进来”，而是下面四件事：

1. 企业数据能否先治理，再给 AI 使用。
2. 不同 AI 系统能否复用同一套治理后的数据资产。
3. 权限、标签、版本、生命周期能否贯穿到索引和查询层。
4. AI 应用消费的是受控数据服务，而不是原始系统直连。

如果这四件事没有做到，平台最终仍然会回到“每个 AI 应用自己拉数据、自己做权限、自己做索引”的重复建设状态。
